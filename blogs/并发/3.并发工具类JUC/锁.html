# 重量级锁

重量级锁是互斥锁。是利用操作系统的同步机制实现的，所以开销比较大。

# 可重入锁和非可重入锁

重入锁ReentrantLock。指的是线程当前已经持有这把锁了，能在不释放这把锁的情况下，再次获取这把锁。

同理，不可重入锁指的是虽然线程当前持有了这把锁，但是如果想再次获取这把锁，也必须要先释放锁后才能再次尝试获取。

# 共享锁和独占锁

共享锁是指同一把锁可以被多个线程同时获得。独占锁就是指一把锁同时只能被一个线程获得。

读写锁就很好地诠释了共享锁和独占锁的理念。读写锁中的读锁，是共享锁，而写锁是独占锁。读锁可以被同时读，可以同时被多个线程持有，而写锁最多只能同时被一个线程持有。

# 公平锁和非公平锁

# 什么是公平和非公平？

公平锁是指按照线程请求的顺序来分配锁。非公平锁是指在一定情况下，线程可以插队。

当然非公平并不能随意插队。假设当前线程在请求获取锁时，恰巧前一个持有这把锁的线程释放了锁，那么当前线程就可以选择插队直接获取到这把锁。反之，如果当前线程请求的时候，锁没有释放，那么这个线程还是会乖乖地进入等待队列。

# 为什么要设置非公平策略

非公平策略是ReentrantLock的默认策略，可见该策略还是很重要的。那么为什么要设置该策略呢？这不是对其他线程不公平？

# 悲观锁和乐观锁

# 悲观锁

字如其名，悲观锁很悲观。它认为不锁住这个资源，别的线程就会来抢。所以它在每次获取修改数据时，都把数据锁住，让其他线程无法访问该数据。这样来确保数据的正确性。

# 乐观锁

乐观锁比较乐观。他认为自己在操作资源的时候，不会有其他线程来干扰。所以它不会锁住被操作的资源。

同时为了确保数据的正确性，在更新资源之前，它会去对比数据，看看在它操作数据的期间，数据有没有被其他线程修改过。

如果没被修改，那就说明真的只有自己在操作，那就正常更新数据。如果发现数据和刚开始拿到的不一样了，它就会认为自己迟了一步，并放弃这次修改，选择重试、报错等其他策略。

# 典型的应用

  • 悲观锁：synchronized关键字和Lock接口

比如Lock接口的实现类ReentrantLock。类中的lock()等方法就是执行加锁，unLock()方法就是执行解锁。

处理资源之前必须先加锁并拿到锁，等到处理完了再解开锁。

  • 乐观锁：原子类

例如 AtomicInteger 在更新数据时，就使用了乐观锁的思想，多个线程可以同时操作同一个原子变量。

  • 又喜又悲：数据库

# 自旋锁和非自旋锁

自旋锁的理念是如果线程现在拿不到锁，并不直接陷入堵塞或者释放CPU资源，而是开始利用循环，不停地尝试获取这把锁。

相反，非自旋锁就是拿不到锁就放弃，去干其他事情。

# 可中断锁和不可中断锁

synchronized 关键字修饰的锁代表的是不可中断锁。一旦线程申请了锁，就没有回头路了。只有等拿到锁以后才能进行其他的逻辑处理。

而ReentrantLock就是一种典型的可中断锁。例如使用lockInterruptibly方法在获取锁的过程中，突然不想获取了，那么也可以在中断之后去做其他的事情，不需要一直傻等到获取到锁才离开。

# Monitor锁

最简单的同步方式就是利用 synchronized 关键字来修饰代码块或者修饰一个方法。而 synchronized 的背后正是利用 monitor 锁实现的。

# Lock接口

# Lock接口简介

Lock接口是Java5引入的。最常见的实现类是ReentrantLock。可以起到锁的作用。

通常情况下，Lock接口只允许一个线程操作共享资源。当然也有例外，比如读写锁可以允许多个线程并发的访问共享资源。

# 常见方法

//方法总览
public interface Lock{
    void lock();
    boolean tryLock();
    boolean trtLock(long time, Timeunit unit) throws InterruptedException;
    void lockInterruptibly() throws InterruptedException;
    void unlock();
}

# 读写锁

读写锁在同一时刻允许多个读线程访问，但是在写线程访问时，所有的读线程和其他写线程均被堵塞。

# 读写锁使用案例

java.util.concurrent.locks.ReadWriteLock

ReadWriteLock接口仅定义了获取读锁和写锁的两个方法。即readLock()和WriteLock方法。

ReentrantReadWriteLock是它的实现类，提供了一些便于外界监控内部工作状态的方法。

public class ReadWriteLockDemo{
    private static final ReentrantReadWriteLock rwl = new 			ReentrantReadWriteLock();
    private static Lock r = rwl.readLock();
    private static Lock w = rwl.writeLock();
}
