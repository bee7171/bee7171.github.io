# 1.Lock接口

  • 锁是一种工具，用来控制对共享资源的访问。
  • Lock是对synchronized的补充，提升。不是代替。
  • Lock接口最常见的实现类是ReentrantLock。

# 1-1.为什么synchronized不够用？

  • **效率低。**锁的释放情况少，试图获得锁时不能设定超时。不能中断一个正在试图获得锁的线程。
  • **不够灵活。**比如读写锁就更加灵活。
  • 无法知道是否成功获取到锁。

# 2.Lock主要方法介绍

  • Lock接口声明了4个方法来获取锁。
    • lock()
    • tryLock()
    • tryLock(long time, TimeUnit unit)
    • lockInterruptibly()

# lock()

  • 最普通的获取锁方法，如果锁已被其他线程获取则等待。

  • Lock不会像synchronized一样在异常时自动释放锁。

  • 最佳解决方式就是使用try-finally，保证发生异常时锁一定会被释放。

  • lock()方法不能被中断，这会带来很大的隐患：一旦陷入死锁，lock()就会陷入无尽的等待。所以这里就会引出tryLock()方法。

# tryLock()

  • 该方法用来尝试获取锁，如果当前锁没有被占用，则获取成功，返回true，反之返回false。
  • 该方法会立即返回，不会拿不到锁就一直在那等。
  • 该方法显然功能更强大，可以根据是否获取到锁来决定程序后续的行为。

# tryLock(long time, TimeUnit unit)

  • 超时就放弃，它会等一段时间。

# lockInterruptibly()

  • 相当于把tryLock(long time, TimeUnit unit)的超时时间设置为无限。在等待锁的过程中，线程可以被中断。

# 3.锁的可见性保证

  • 遵守happens-before原则
  • Lock的加解锁和synchronized有同样的内存语义。也就是说，下一个线程加锁后能看到所有前一个线程解锁前发生的所有事情。

# 4.锁的分类

  • 这些分类是从不同角度去看的，意思就是并非互斥的，一把锁可能属于多个分类的锁。

# 3-1.悲观锁和乐观锁

  • 悲观锁的劣势：

    • 阻塞和唤醒带来的性能问题。
    • 永久阻塞：持有锁的线程如果永久阻塞，那么其他线程永远得不到执行。
    • 优先级反转：低优先级的线程拿到锁以后，如果一直不释放。优先级就错乱了。

  • 悲观锁：

    • 例子：Lock接口和synchronized关键字

  • 乐观锁：

    • 认为自己操作时不会有其他线程来干扰，所以并不会锁住被操作对象。

    • 更新的时候，会去对比在修改期间数据有没有被其他线程动过，如果没被改变，就正常修改数据。

    • 如果数据被改变，就会采取报错，放弃等策略。

    • 用CAS算法来实现。

    • 例子：原子类和并发容器等。

      乐观锁的典型例子：git

往远端仓库push的时候，会检查本地仓库和远程仓库的版本号是否一致。不一样就代表有人修改过了，这次提交就失败。

  • 两种锁的各自使用场景：

    • **悲观锁：**适合并发写入多的情况。适用于临界区持有锁时间比较长的情况，可以避免大量无用的自旋等消耗。
      • 1.临界区有IO操作。
      • 2.临界区代码复杂或者循环量大。
      • 3.临界区竞争非常激烈。
    • **乐观锁：**适合并发写入少，大多都是读取的场景。能让读取性能大大提高。

# 3-3.公平锁与非公平锁

  • 公平指的是完全按照线程请求的顺序来获取锁。非公平是指在一定情况下，可以插队。（合适的时机插队）

  • 这样设计的目的是为了提高效率。避免唤醒带来的空档期。

    比如有3个线程A，B，C

A持有锁，A要释放锁，B就从阻塞状态中唤醒，唤醒是有开销和耗时的。
而C这是突然来请求锁，它一直处于唤醒状态。
这是双赢的局面，C请求到了,B唤醒也是耗时的，等C释放了，B正好完全唤醒。

可以有效得提高吞吐量。

# 3-4.共享锁和独占锁

  • 共享锁又称为读锁，可以查看但无法修改或删除。可以多个线程同时获取到读锁。
  • 典型案例：ReentrantReadWriteLock。读锁是共享锁，写锁是独占锁。
