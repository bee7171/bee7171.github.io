# 1.ThreadLocal的使用场景

  • 典型场景1：
    • 每个线程需要一个独享的对象。（通常是工具类。比如SimpleDateFormat和Random。）
  • 典型场景2：
    • 每个线程需要保存全局变量。（例如在拦截器中获取用户信息）可以让不同方法直接使用，避免参数传递的麻烦。

# 1-1. 典型场景1

  • 每个线程内有自己的实例副本，不共享。比如教材只有一本，大家一起做笔记会有安全问题，复印好多份就没问题。

代码演示：SimpleDateFormat

//两个线程分别使用SimpleDateFormat打印日期

public class ThreadLocalNormalUsage00 {
    public static void main(String[] args) {
        new Thread(new Runnable() {
            @Override
            public void run() {
                String date = new ThreadLocalNormalUsage00().date(10);
                System.out.println(date);
            }
        }).start();

        new Thread(new Runnable() {
            @Override
            public void run() {
                String date = new ThreadLocalNormalUsage00().date(104707);
                System.out.println(date);
            }
        }).start();
    }

    public String date(int seconds){
         //单位是毫秒，从1970.1.1 00:00:00开始计时
         Date date = new Date(1000 * seconds);
         SimpleDateFormat simpleDateFormat = new SimpleDateFormat("yyyy-MM-dd hh:mm:ss");
         return simpleDateFormat.format(date);
     }
}

import java.text.SimpleDateFormat;
import java.util.Date;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class ThreadLocalNormalUsage04 {
    public static ExecutorService threadPool = Executors.newFixedThreadPool(10);

    static SimpleDateFormat simpleDateFormat = new SimpleDateFormat("yyyy-MM-dd hh:mm:ss");
    public static void main(String[] args) {
        for (int i = 0; i < 1000; i++) {
            int finalI = i;
            threadPool.submit(new Runnable() {
                @Override
                public void run() {
                    String date = new ThreadLocalNormalUsage04().date(finalI);

    public String date(int seconds){
         //单位是毫秒，从1970.1.1 00:00:00开始计时
         Date date = new Date(1000 * seconds);
         String s = null;
         synchronized (ThreadLocalNormalUsage04.class){
             s = simpleDateFormat.format(date);
         }
         return s;
     }
}

  • 总结：SimpleDateFormat的进化之路
1. 2个线程分别使用自己的SimpleDateFormat,这没问题。
2. 后来变成10个，就需要10个线程和10个SimpleDateFormat对象，这不优雅。
3. 当变成1000个时，必须使用线程池。（否则内存消耗太多）
4. 所有的线程共用一个SimpleDateFormat对象，结果发生了线程安全问题
5. 给对象加锁，这可以解决。但是线程一个个排队，效率也太低了。

  • 更好的解决方案——ThreadLocal

图中，我们用线程池中的10个线程来执行1000个任务。我们在10个线程里分别使用一个SimpleDateFormat对象。这样既能避免锁带来的性能损耗，又能高效率的执行问题。还保证了线程安全问题。


public class ThreadLocalNormalUsage05 {
    public static ExecutorService threadPool = Executors.newFixedThreadPool(10);


    public static void main(String[] args) {
        for (int i = 0; i < 1000; i++) {
            int finalI = i;
            threadPool.submit(new Runnable() {
                @Override
                public void run() {
                    String date = new ThreadLocalNormalUsage05().date(finalI);
                    System.out.println(date);
                }
            });
        }
        threadPool.shutdown();

    }

    public String date(int seconds){
         //单位是毫秒，从1970.1.1 00:00:00开始计时
         Date date = new Date(1000 * seconds);
         //SimpleDateFormat simpleDateFormat = new SimpleDateFormat("yyyy-MM-dd hh:mm:ss");
        
        SimpleDateFormat simpleDateFormat = ThreadSafeFormatter.dateFormatThreadLocal.get();
        return simpleDateFormat.format(date);
    }
}

//使用ThreadLocal方法来创建实例副本
class ThreadSafeFormatter{
    public static ThreadLocal<SimpleDateFormat> dateFormatThreadLocal = new ThreadLocal<>(){
        //重写该方法，这个方法是用来初始化用的
        @Override
        protected SimpleDateFormat initialValue() {
            return new SimpleDateFormat("yyyy-MM-dd hh:mm:ss");
        }
    };

    //第二种写法：使用lambda表达式
    public static ThreadLocal<SimpleDateFormat> dateFormatThreadLocal2 = ThreadLocal
            .withInitial(() -> new SimpleDateFormat("yyyy-MM-dd hh:mm:ss"));
}

# 1-2.典型场景2

  • 每个线程内需要保存全局变量，可以共享给所有方法使用，避免参数传递的麻烦。

**例子：**当前用户信息需要被线程内所有方法共享

图中的解决方案是将user作为参数层层传递。从方法1传递到方法2。。。。。等等。这样会导致代码冗余不易维护。

解决方法：

  • 用ThreadLocal保存一些业务内容（用户权限信息，从用户系统得到的id等信息等）。每一个请求线程内部的user对象是不同的，所以用static变量是不可以的。
  • 这些信息在同一个线程内相同，但是在不同线程间的用户信息是不相同的。
  • 在线程生命周期内，都通过这个静态ThreadLocal实例的get方法来获得对象。避免了将这个对象作为参数传递的麻烦。

用法：

  • 强调的是同一个请求（同一个线程）内不同方法间的共享。
  • 不需要重写initialValue方法，必须手动调用set方法。

# 2.ThreadLocal的两个作用（总结）

  • 让某个需要用到的对象在线程间隔离。（每个线程都有自己独立的对象）
  • 在任何方法中都可以轻松地获取该对象。

场景1：initialValue

​ 在ThreadLocal第一次get的时候把对象给初始化出来，对象的初始化时机由我们控制。

场景2：set

​ 保存到ThreadLocal里的对象的生成时机不由我们随意控制，例如拦截器生成的用户信息。用ThreadLocal.set直接放到我们的ThreadLocal中，以便后续使用。

# 3.ThreadLocal的好处

  • 达到线程安全
  • 不需要加锁，提高执行效率
  • 更高效的利用内存，节省开销
  • 避免了参数传递的麻烦。使得代码耦合度更低，更优雅。

# 4.原理，源码分析

  • 搞清楚Thread、ThreadLocalMap、ThreadLocal的关系

一个Thread对应一个ThreadLocalMap，Map里有多个ThreadLocal对象，用键值对的方式存储。

# 4-1.initialValue方法

  • 该方法会返回当前线程对应的初始值，这是一个延迟加载的方法，只有调用get方法才会触发。
  • 当线程第一次调用get方法访问变量时，将调用此方法。除非线程先前调用了set方法，在这种情况下，不会为线程调用本initialValue方法。
  • 每个线程最多调用一次get方法，如果已经调用remove了，则可以再次调用此方法。
  • 如果部重写该方法，这个方法会返回null。一般使用匿名内部类的方法来重写。以便在后续使用中可以初始化副本对象。

# 4-2.set方法

  • 为这个线程设置一个新值。

# 4-3.get方法

  • 先取出当前线程的ThreadLocalMap。
  • 调用map.getEntry方法，把本ThreadLocal的引用作为参数传入。
  • 取出map中属于本ThreadLocal的value。

# 4-4.remove方法
